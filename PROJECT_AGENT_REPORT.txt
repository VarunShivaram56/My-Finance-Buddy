============================================================
1) PROJECT IDENTITY
============================================================
Project type:
- Local-first AI-based personal finance assistant.
- Input: bank statement PDF (redacted).
- Output: parsed transactions, dashboard analytics, category tagging, AI insight bullets, finance chatbot (RAG/general).

Primary architecture:
- Hybrid pipeline: Rule-based parsing/categorization first, LLM fallback only for uncertain rows.
- Backend: FastAPI + SQLAlchemy + PDF parsing + Chroma vector index + Groq-compatible chat API.
- Frontend: React + Tailwind + Recharts + Axios.

Core objective:
- Keep deterministic finance processing strong and only invoke LLM where needed.

6) INGESTION PIPELINE (UPLOAD TO STORAGE)
============================================================
Main orchestrator:
- backend/services/statement_service.py

Detailed sequence:
1. Validate PDF upload and read bytes.
2. Hash bytes and check duplicate uploads for the same user.
3. Extract rows:
   - Uses bank-specific extractor family first:
     - standard_ledger (Karnataka Bank)
     - canara_ledger
     - sbi_ledger
   - Falls back to generic table/text extraction if specialized extraction yields nothing.
4. Normalize extracted rows to text lines.
5. Parse each row via deterministic parser:
   - date, amount, transaction type, merchant, confidence.
6. Low-confidence rows (confidence < LLM_CONFIDENCE_THRESHOLD) are sent to LLM enrichment.
7. Build valid transaction set (requires date, merchant, amount, type).
8. Categorize each valid transaction through hybrid categorizer.
9. Persist statement + transactions to DB.
10. Generate optional AI insights summary.
11. Rebuild user RAG index in Chroma.
12. Return dashboard summary payload.

============================================================
7) RULE-BASED VS LLM LOGIC (HYBRID DESIGN)
============================================================
Rule-first behavior:
- Parsing:
  - Regex/date patterns and amount extraction.
  - Transaction type inferred by keywords + DR/CR hints.
  - Merchant extraction heuristics for UPI and noisy statement text.
- Categorization:
  - Existing user historical category reuse by normalized description signature.
  - Personal transfer heuristic to avoid false merchant categories.
  - Merchant keyword map (static).
  - Category keyword groups.

LLM fallback behavior:
- Parser fallback:
  - Only low-confidence rows are enriched by Agent 1.
- Categorization fallback:
  - Only unresolved descriptions go to Agent 2.
- Insight generation:
  - Agent 3 optional, bounded by max transaction thresholds.

Failure behavior:
- If LLM unavailable or errors:
  - Parsing and categorization fall back to safe defaults.
  - App still works using deterministic logic.

Configured API keys in CURRENT backend/.env:

- AGENT_TWO_API_KEY
- AGENT_THREE_API_KEY

Agent responsibilities:
- Agent 1 (enrich_transactions):
  - Purpose: convert ambiguous raw statement rows into strict JSON objects.
  - Called only for low-confidence parser rows.
  - Uses dedicated GroqClient instance with AGENT_ONE_API_KEY.
  - Includes strict JSON retry if the first response is not parseable JSON.
- Agent 2 (categorize_batch):
  - Purpose: classify unresolved transaction descriptions into fixed category list.
  - Called only for unresolved items after all deterministic checks.
  - Uses dedicated GroqClient instance with AGENT_TWO_API_KEY.
- Agent 3 (summarize_anomalies and chat):
  - Purpose: dashboard AI insight bullets and chatbot response generation.
  - Uses dedicated GroqClient instance with AGENT_THREE_API_KEY.
  - Used for:
    - summarize_anomalies (insight bullets)
    - /chat in both rag and general modes

Embedding behavior:
- Current RAG embedding path is local deterministic hashed semantic embedding fallback.
- No external embedding provider is required for the current code path.

Rate-limit handling:
- GroqClient normalizes quota/limit errors to:
  - "API call limit reached, change the api key."

Operational note:
- OpenRouter has been removed from live source usage.
- backend/services/openrouter_client.py is deleted.

11) CONFIG FLAGS WITH BEHAVIOR IMPACT
============================================================
Critical flags:
- AGENT_ONE_API_KEY / AGENT_TWO_API_KEY / AGENT_THREE_API_KEY:
  - Missing key for an agent disables that agent's external LLM calls and falls back safely.
- LLM_CONFIDENCE_THRESHOLD:
  - Lower means fewer rows sent to Agent 1.
  - Higher means more LLM enrichment calls.
- LLM_BATCH_SIZE:
  - Controls chunk size for enrichment/categorization batching.
- ENABLE_AI_INSIGHTS:
  - Enables/disables Agent 3 insight bullet generation.
- ENABLE_VECTOR_MEMORY:
  - Currently false and vector memory module is not wired into live categorizer path.
- MAX_INSIGHT_TRANSACTIONS / MAX_AI_INSIGHT_STATEMENT_SIZE:
  - Guardrails for insight calls.
