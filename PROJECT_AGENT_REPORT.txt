============================================================
1) PROJECT IDENTITY
============================================================
Project type:
- Local-first AI-based personal finance assistant.
- Input: bank statement PDF (redacted) + manual transactions + loans/assets.
- Output: parsed transactions, dashboard analytics, category tagging, AI insight bullets, finance chatbot (SQL-RAG + general).

Primary architecture:
- Hybrid pipeline: deterministic parsing/categorization first, LLM fallback only for uncertain rows.
- Backend: FastAPI + SQLAlchemy + PDF parsing + SQL-based RAG (text-to-SQL) + Groq/OpenRouter chat clients.
- Frontend: React + Tailwind + Recharts + Axios.

Core objective:
- Keep deterministic finance processing strong and only invoke LLMs where needed.

6) INGESTION PIPELINE (UPLOAD TO STORAGE)
============================================================
Main orchestrator:
- backend/services/statement_service.py

Detailed sequence:
1. Validate PDF upload and read bytes.
2. Hash bytes and check duplicate uploads for the same user.
3. Extract rows:
   - Uses bank-specific extractor family first:
     - standard_ledger (Karnataka Bank)
     - canara_ledger
     - sbi_ledger
   - Falls back to generic table/text extraction if specialized extraction yields nothing.
4. Normalize extracted rows to text lines.
5. Parse each row via deterministic parser:
   - date, amount, transaction type, merchant, confidence.
6. Low-confidence rows are sent to LLM enrichment (Agent 1).
7. Build valid transaction set (requires date, merchant, amount, type).
8. Categorize each valid transaction through hybrid categorizer (rule-first + LLM fallback).
9. Persist statement + transactions to DB.
10. Generate optional AI insights summary (Agent 3).
11. Refresh cached dashboard payloads (no vector index rebuild; SQL-RAG reads live DB).

============================================================
7) RULE-BASED VS LLM LOGIC (HYBRID DESIGN)
============================================================
Rule-first behavior:
- Parsing:
  - Regex/date patterns and amount extraction.
  - Transaction type inferred by keywords + DR/CR hints.
  - Merchant extraction heuristics for UPI and noisy statement text.
- Categorization:
  - Existing user historical category reuse by normalized description signature.
  - Personal transfer heuristic to avoid false merchant categories.
  - Merchant keyword map (static).
  - Category keyword groups.

LLM fallback behavior:
- Parser fallback:
  - Only low-confidence rows are enriched by Agent 1.
- Categorization fallback:
  - Only unresolved descriptions go to Agent 2.
- Insight generation:
  - Agent 3 optional, bounded by max transaction thresholds.

Failure behavior:
- If LLM unavailable or errors:
  - Parsing and categorization fall back to safe defaults.
  - App still works using deterministic logic.

8) LLM PROVIDERS AND MODEL-TO-AGENT MAPPING
============================================================
Providers:
- Groq (Agent 1 + Agent 2): https://api.groq.com/openai/v1
- OpenRouter (Agent 3): https://openrouter.ai/api/v1

Configured model keys (backend/.env or env vars):
- AGENT_ONE_API_KEY (Groq)
- AGENT_TWO_API_KEY (Groq)
- OPENROUTER_API_KEY or AGENT_THREE_API_KEY (OpenRouter)

Configured models (defaults in backend/utils/config.py):
- AGENT_ONE_MODEL=groq/compound
- AGENT_TWO_MODEL=groq/compound
- AGENT_THREE_MODEL=openai/gpt-oss-120b
- EMBEDDING_MODEL=nomic-embed-text (legacy; not used by SQL-RAG)

Agent responsibilities:
- Agent 1 (enrich_transactions, Groq):
  - Convert ambiguous raw statement rows into strict JSON objects.
  - Called only for low-confidence parser rows.
  - Strict JSON retry if first response is not parseable JSON.
- Agent 2 (categorize_batch, Groq):
  - Classify unresolved transaction descriptions into fixed category list.
  - Called only after deterministic checks.
- Agent 3 (OpenRouter):
  - Used for dashboard AI insight bullets.
  - Used for SQL-RAG text-to-SQL + synthesis responses.
  - Used for general chat mode.

Rate-limit handling:
- Both clients normalize quota/limit errors to:
  - "API call limit reached, change the api key."

9) RAG AND CHAT DESIGN (SQL-BASED)
============================================================
Core files:
- backend/rag/chatbot_service.py
- backend/rag/prompt_builder.py

RAG behavior:
- No Chroma or vector DB in the active path.
- RAG mode generates a SQL SELECT query against the live DB, then synthesizes a plain-text answer.
- Section-aware schema:
  - section=transactions: statements + transactions + non_banking_transactions
  - section=loans_and_liabilities: loans + assets
- Strict guardrails:
  - Only SELECT queries allowed.
  - Result sets limited to 20 rows for multi-row outputs.
  - Enforces user_id filtering for security (transactions must join statements on user_id).

General chat:
- A standard OpenRouter chat completion without DB grounding.

11) CONFIG FLAGS WITH BEHAVIOR IMPACT
============================================================
Critical flags:
- AGENT_ONE_API_KEY / AGENT_TWO_API_KEY (Groq):
  - Missing key disables Agent 1/2 external LLM calls and falls back safely.
- OPENROUTER_API_KEY (OpenRouter):
  - Missing key disables Agent 3 (RAG + general chat + insight bullets).
- LLM_CONFIDENCE_THRESHOLD:
  - Lower means fewer rows sent to Agent 1.
  - Higher means more LLM enrichment calls.
- LLM_BATCH_SIZE:
  - Controls chunk size for enrichment/categorization batching.
- ENABLE_AI_INSIGHTS:
  - Enables/disables Agent 3 insight bullet generation.
- ENABLE_VECTOR_MEMORY:
  - Currently false and vector memory module is not wired into live categorizer path.
- MAX_INSIGHT_TRANSACTIONS / MAX_AI_INSIGHT_STATEMENT_SIZE:
  - Guardrails for insight calls.
