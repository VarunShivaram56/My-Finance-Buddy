1. PROJECT OVERVIEW
============================================================
My Finance Buddy is a local-first, AI-augmented full-stack web application designed for personal finance management.
Problem Modeled: Scattered financial records, loans, and assets.
Solution provided: An automated pipeline that ingests bank PDFs, securely extracts transactions locally, dynamically parses unknown elements via LLM agents to ensure privacy, and visualizes them on a unified React dashboard.

4. DETAILED DATABASE SCHEMA (SQLITE3)
============================================================
The system runs strictly on SQLite3. AI Agents MUST respect these exact table names, columns, and types. Do NOT assume MySQL commands (e.g. use `STRFTIME` instead of `YEAR()`). All foreign keys point heavily to `users.id`.

Tables:

1. `users` -> Primary account holders
   - `id` (INTEGER PPI)
   - `name` (VARCHAR)
   - `user_id` (VARCHAR UNIQUE)
   - `password_hash`, `password_salt` (VARCHAR)
   - `created_at` (DATETIME)

2. `user_sessions` -> Bearer token persistence
   - `id` (INTEGER PPI)
   - `user_id` (FOREIGN KEY -> users.id)
   - `token_hash` (VARCHAR UNIQUE)
   - `created_at`, `last_used_at` (DATETIME)

3. `statements` -> Tracks uploaded PDFs to prevent duplication
   - `id` (INTEGER PPI)
   - `user_id` (FOREIGN KEY -> users.id)
   - `filename` (VARCHAR)
   - `uploaded_at` (DATETIME)
   - `source_hash` (VARCHAR) -> SHA256 of the PDF bytes.

4. `raw_statement_lines` -> Stores raw unparsed lines from PDF

   - `statement_id` (FOREIGN KEY -> statements.id)
   - `raw_text` (TEXT)
   - `parser_confidence` (FLOAT)

5. `transactions` -> The central ledger holding bank rows

7. `loans`

5. AI AGENT PIPELINE & WORKFLOWS
============================================================
The system avoids expensive massive context dumps and 413 Errors by creating a multi-layered verification system. All AI Agents are built natively using the `GroqClient` routing to `llama-3.1-8b-instant`. The single API key variable `GROQ_API_KEY` operates all agents.

Step-by-Step PDF Ingestion (`upload_router.py` -> `statement_service.py`):
1. User uploads a PDF. Python `pdfplumber` rips all text.
2. The custom `parsing.py` regex module slices the text into raw rows and scores it dynamically on confidence.
3. Agent 1 (Merchant Extraction): 
   - ONLY triggers on transactions where the deterministic parser failed to confidently find a merchant name (confidence score falloff). 
   - Prompts the LLM purely for the merchant name using batched sets.
   - Batch limits (e.g. 10 rows) logic is strictly enforced using `time.sleep()` injected inside `agent_manager.py` to prevent Groq API Tier 1 limits (6,000 TPM/30 RPM).
4. Deterministic Categorization (`categorizer.py`):
   - Fast evaluation against historical identical database merchants.
   - Checks static regex maps in `utils/merchant_rules.py` (e.g. "zomato" -> "Food & Dining").
5. Agent 2 (Categorization LLM):
   - Only processes the scraps completely missed by deterministic filtering. Returns exactly one category from the predefined string list.
   - Has strict Token limiting.

6. RAG CHATBOT ARCHITECTURE (AGENT 3) 
============================================================
Housed in `rag/chatbot_service.py`. It is a true Text-to-SQLite3 RAG paradigm. We do NOT use ChromaDB or Vector stores.

Flow:
1. User submits natural text ("How much money did I spend on food?").
2. The Backend detects the user context, injects the SQLite schemas natively into the prompt.
3. It instructs Agent 3 to generate mathematically robust aggregated `SQLite3` queries strictly (no markdown blocks allowed, favors `SUM()` and `STRFTIME` functions, and uses `LIMIT 15` on raw row pulls to prevent TPM crash).
4. Query runs against the DB locally.
  - Sub-Flow (Retry Loop): If SQL validation fails, SQLAlchemy throws an exception. The backend automatically captures the string error code, feeds the broken SQL and the Error back to the LLM urging a fix, and tries again.
5. Resulting DB JSON Array is returned to Agent 3 for Synthesis mode.
6. The Synthesis Agent formats the data strictly using Indian Rupees (`Rs. `) constraint flags, discards markdown tables, and creates a concise bulleted answer delivered directly to the React Chat GUI.
